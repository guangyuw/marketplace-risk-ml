# Marketplace Risk Scoring — Production ML Pipeline

End-to-end **machine learning pipeline** for scoring marketplace transaction risk and routing low-risk orders to auto-approval. Built to demonstrate production ML patterns relevant to marketplace businesses (fraud/risk, seller quality, buyer trust).

**Classical ML for the decision; an LLM only for de-identified review assist.** The calibrated model owns routing (auto-approve / manual-review / audit). For flagged transactions, an optional LLM layer drafts a structured, policy-bounded review brief for the human reviewer — with an evaluation harness and a deterministic offline fallback. See [`docs/llm_assist.md`](docs/llm_assist.md).

> **Note:** Uses synthetic data generated to mirror real marketplace transaction patterns. The architecture, feature design, and serving patterns reflect production-grade approaches.

---

## Business problem

| This project | Marketplace use case |
|---|---|
| `is_high_risk` score | Seller/buyer fraud, bad listing, churn risk |
| Auto-approve below threshold | Fast-track trusted low-risk transactions |
| False-clear rate cap | Business tolerance for mistakes |
| PSI monitoring | Detect drift when new events/venues launch |
| Random audit sample | Unbiased labels despite selective approval |

## Architecture

```
Snowflake SQL (features)  →  Train (MLflow)  →  Model Registry  →  FastAPI serve
                                      ↓
                              PSI / threshold / audit routing
```

## Quick start

```bash
cd marketplace-risk-ml
python -m venv .venv && source .venv/bin/activate

# production dependencies
pip install -r requirements.txt

# 1) Train + log to MLflow + save artifacts
python -m src.train

# 2) Run tests (requires dev dependencies)
pip install -r requirements-dev.txt
pytest -q

# 3) Start API
python -m src.serve
# or: uvicorn src.serve:app --reload --port 8000
```

**Sample request**

```bash
curl -s -X POST http://localhost:8000/predict \
  -H "Content-Type: application/json" \
  -d '{
    "ticket_price": 185.0,
    "quantity": 2,
    "buyer_age": 29,
    "venue_type": "arena",
    "event_category": "concert",
    "section_code": "S0042",
    "seller_prior_dispute_rate": 0.04,
    "buyer_prior_purchases": 3,
    "event_date": "2025-03-15"
  }' | python -m json.tool
```

**Review-assist request** (structured brief for a flagged transaction)

```bash
curl -s -X POST http://localhost:8000/review-assist \
  -H "Content-Type: application/json" \
  -d '{
    "ticket_price": 1400.0,
    "quantity": 8,
    "buyer_age": 22,
    "venue_type": "festival",
    "event_category": "concert",
    "section_code": "S0003",
    "seller_prior_dispute_rate": 0.42,
    "buyer_prior_purchases": 0,
    "event_date": "2025-07-20"
  }' | python -m json.tool
```

Returns the de-identified evidence + a policy-bounded brief. Auto-approved
transactions return `assist_applicable: false` (no brief). With no
`OPENAI_API_KEY` set, the brief is a deterministic fallback. Evaluate the layer
with `python -m evals.run_eval`. Details in [`docs/llm_assist.md`](docs/llm_assist.md).

**Docker**

```bash
python -m src.train   # must run first to create artifacts/
docker build -t marketplace-risk-ml .
docker run -p 8000:8000 marketplace-risk-ml
```

## Production-minded design choices

1. **Temporal split** — train on past, evaluate on future (not random split).
2. **Threshold = business decision** — maximize automation subject to false-clear tolerance (not default 0.5).
3. **Frequency encoding** — high-cardinality categories without target leakage.
4. **Versioned bundle** — model + freq_maps + threshold persisted together so serving never drifts from training.
5. **MLflow** — experiment tracking, metrics, model artifacts, optional registry.
6. **Serving** — containerized FastAPI with health check; `model` + `freq_maps` + `threshold` loaded from a single bundle.
7. **Monitoring** — PSI on scores/features; random audit routing to prevent selective-label bias in retraining.
8. **Probability calibration** — Brier score and calibration curve logged per run.
9. **LLM as assist, not decision** — the LLM review layer only drafts a policy-bounded brief for flagged transactions, on a de-identified payload (no PII), off the critical path (deterministic fallback), and is evaluated for hallucination/recall/faithfulness. See [`docs/llm_assist.md`](docs/llm_assist.md).

## Snowflake & Databricks

- **Snowflake:** `sql/feature_extraction_snowflake.sql` implements leakage-safe window features using `ROWS BETWEEN UNBOUNDED PRECEDING AND 1 PRECEDING`. Training consumes a point-in-time snapshot.
- **Databricks:** Port `src/train.py` to a notebook; use Databricks-hosted MLflow. See `docs/DATABRICKS.md` for the integration walkthrough.
